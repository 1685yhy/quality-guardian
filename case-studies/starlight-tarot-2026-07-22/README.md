# Quality Guardian 案例: 星光映照塔罗小程序

## 案例信息

| 项 | 值 |
|---|---|
| 项目 | 星光映照 — AI塔罗牌占卜微信小程序 |
| 验收时间 | 2026-07-22 |
| 验收阶段 | 阶段三: 开发后盲测验收 |
| 项目规模 | 11页面 + 3组件 + 后端17端点 + 8数据表 |
| 代码量 | 前端~8000行 + 后端~3000行 |
| 验收方式 | 代码级盲测 (小程序无URL，适配为代码审查) |

## Quality Guardian 执行方式

### 调度架构

```
Orchestrator (主调度器)
├── Guardian 团队 (6维并行)
│   ├── 可达性 (Reachability)
│   ├── 可理解性 (Understandability)
│   ├── 可靠性 (Reliability)
│   ├── 响应性 (Responsiveness)
│   ├── 愉悦性 (Delight)
│   └── 包容性 (Inclusivity)
└── Simulator 团队 (4画像并行)
    ├── 小雨 - 新手小白 (25岁, 第一次接触塔罗)
    ├── Luna - 效率专家 (28岁, 塔罗3年经验)
    ├── 阿杰 - 暴躁用户 (30岁, 程序员, 极不耐烦)
    └── Vivian - 完美主义 (32岁, UI设计师)
```

### 执行参数
- **10个Agent全并行派发**，独立上下文，盲测隔离
- **每个Guardian** 传入该维度的检查方法论 + 项目文件读取权限
- **每个Simulator** 传入用户画像 + 核心页面代码
- **不传入** 任何项目开发历史、设计文档、API文档

## 验收结果

### 总体: 63/100 ❌

| 维度 | 得分 | 状态 |
|------|------|------|
| 愉悦性 | 95% | ✅ 通过 |
| 响应性 | 94% | ✅ 通过 |
| 可达性 | 65% | ❌ 不通过 |
| 可理解性 | 47% | ❌ 不通过 |
| 可靠性 | 44% | ❌ 不通过 |
| 包容性 | 43% | ❌ 不通过 |

### 发现的 P0 问题

1. **定价欺诈风险** — 前端显示价格与后端实际扣款不一致
2. **JWT密钥默认值** — `"change-me-in-production"` 未替换
3. **无障碍完全缺失** — 盲人用户无法使用核心功能
4. **会员入口严重不足** — 首页无升级路径

### 4位模拟用户的真实反馈

| 小雨 (新手) | "被认真对待了，画面惊艳，但牌阵术语看不懂" |
| Luna (专家) | "卡牌数据深度顶级，但核心牌阵锁付费墙会赶走真用户" |
| 阿杰 (暴躁) | "3分钟5次想关，默认沉浸模式是强制的等待税" |
| Vivian (设计师) | "设计系统A-，执行B，会员页覆写全局类名不可接受" |

## 对 Quality Guardian 框架的反馈

### 做得好的
- **盲测隔离有效**: 10个Agent在不知道彼此结论的情况下，多个Agent独立发现了相同问题（如术语混乱被可理解性Guardian和小雨、Luna三个Agent同时发现），说明框架的交叉验证机制有效
- **Simulator > Guardian 的裁决规则合理**: 响应性Guardian对3段加载评了3分，但阿杰的反馈揭示了"默认沉浸模式对暴躁用户是劝退"——真实用户反馈修正了理论评分
- **发现了开发者视角的盲区**: 定价不一致、会员表数据矛盾这类问题，开发者自己测试时不太会注意到，但Guardian从用户视角独立审查能发现

### 可改进的
- **小程序适配**: 框架设计为URL驱动(L1)，小程序只能用代码审查(L2/L3)，真实度打折。建议增加微信开发者工具截图/录屏的自动采集能力
- **后端盲测困难**: 后端代码审查难以做到真正的"盲测"——Guardian能看出代码写了什么保护，但无法模拟真实攻击
- **国际化**: 当前框架和输出都是中文，若面向国际化项目需要英文版本
