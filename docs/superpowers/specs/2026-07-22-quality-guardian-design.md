# Quality Guardian — 全维度品质保障框架

**版本**: v1.0  
**日期**: 2026-07-22  
**作者**: 闫海洋 (PM, 好未来)  
**定位**: 开源 GitHub 项目，为任何软件项目提供 AI 驱动的验收测试 + 真实用户模拟能力

---

## 1. 项目背景与动机

### 1.1 痛点

- **开发者自验盲区**: 自己做的项目自己验收，因熟悉实现细节而忽略体验问题
- **验收标准缺失**: 没有前置的验收标准，验收时往往随机抽查，覆盖不全
- **维度单一**: 传统测试只关注功能正确性，忽略设计、交互、动效、文案、性能等用户可感知的一切维度
- **缺乏真实用户视角**: 内部验收通过 ≠ 真实用户觉得好用。缺少"像真实用户一样使用产品并反馈"的能力
- **跨项目类型困难**: 不同产品类型（Web/小程序/App/游戏）验收标准完全不同，没有统一框架

### 1.2 愿景

打造一个"丢进任何项目就能用"的品质保障框架——像一支专业的 QA 团队 + 一群真实用户，自动识别项目类型，自动生成验收标准，盲测验收 + 真实用户模拟，全流程覆盖。

## 2. 核心设计理念

### 2.1 通用性来自推理能力，不是穷举清单

```
❌ 错误思路: 穷举所有产品类型的检查清单
   web-app-checklist.md / game-rpg-checklist.md / mini-program-checklist.md ...
   → 永远做不完，永远跟不上新产品形态

✅ 正确思路: 教 Agent 怎么思考，不是教它检查什么
   1. 读取项目 → 理解"这是什么产品"
   2. 套用通用品质框架 → 推理"这个产品应该检查什么"
   3. 生成匹配的用户画像 → "谁会用它"
   4. 选择匹配的模拟方式 → "怎么用它"
```

### 2.2 两大核心系统

| | **Quality Guardian（验收管家）** | **User Simulator（真实用户模拟）** |
|---|---|---|
| 角色 | 质检员，拿标准对照打分 | 真实用户，自然使用产品 |
| 视角 | "这个按钮对齐了吗？" | "这个按钮我怎么找不到？" |
| 产出 | 通过/不通过 + 问题清单 | 体验日记 + 困惑点 + 情绪曲线 |
| 方式 | 结构化检查清单，逐项验证 | 自由探索式使用，自然反馈 |

两者互补：Guardian 保证底线，Simulator 探索上限。

## 3. 通用品质框架（6 维度体系）

任何产品，无论类型，都可以被这 6 个维度覆盖。Agent 拿到项目后，基于这 6 个维度**推理**出该产品的具体检查项。

| 维度 | 核心问题 | Web 管理后台示例 | 游戏示例 | 小程序示例 |
|------|---------|----------------|---------|-----------|
| **可达性** Reachability | 用户能不能找到、能不能用上？ | 菜单层级是否合理、搜索是否有效 | 新手引导是否清晰、核心玩法入口是否明显 | 首页入口是否明显、扫码是否顺畅 |
| **可理解性** Understandability | 用户知不知道这是什么、怎么用？ | 表格字段含义是否清晰、操作是否有说明 | 数值面板是否好懂、任务目标是否明确 | 功能介绍是否明白、文案是否说人话 |
| **可靠性** Reliability | 用了会不会出错、崩了会不会丢？ | 数据提交是否准确、异常是否妥善处理 | 存档是否安全、网络断开是否保护进度 | 支付是否可靠、授权是否合规 |
| **响应性** Responsiveness | 操作有没有反馈、快不快？ | 列表加载速度、按钮点击反馈 | 战斗手感是否跟手、UI 过渡是否流畅 | 页面切换流畅度、下拉刷新反馈 |
| **愉悦性** Delight | 用着开不开心、有没有惊喜？ | 视觉是否清爽、微交互是否精致 | 特效/音效/打击感/动画品质 | 动效是否精致、情感化设计 |
| **包容性** Inclusivity | 不同人、不同环境能不能用？ | 屏幕尺寸适配、键盘导航支持 | 色盲模式、难度选择、语言本地化 | 老年模式、弱网环境、不同机型 |

## 5. 全流程设计

### 5.1 阶段一：开发前 — 验收标准生成

```
输入: 项目仓库（需求文档/PRD/设计稿）
     ↓
orchestrator 读取项目，推理产品本质
     ↓
输出:
  1. 产品类型判断 + 6 维度映射
  2. 验收标准清单（每个维度 5-15 条具体检查项）
  3. 用户画像 × 3-5 个
  4. 关键场景矩阵（正常流程 + 异常流程 + 边界条件）
```

**关键**: 这个阶段 Agent **不读实现代码**，只读需求和设计文档，确保验收标准是"用户视角的"，而非"开发者视角的"。

### 5.2 阶段二：开发中 — 持续检查

```
输入: 开发中的产品（可交互版本/dev 环境）
     ↓
按节奏触发:
  - 每完成一个模块 → Guardian 对已完成部分做增量检查
  - 每完成一个功能 → Simulator 以 1-2 个用户画像做体验试跑
     ↓
输出:
  - 实时问题清单（即时反馈，不等全部做完）
  - 开发中的体验风险预警
```

### 5.3 阶段三：开发后 — 盲测验收

```
输入: 完整产品（staging/预发布环境）
     ↓
Guardian 团队 (并行):
  每个 Guardian Agent 独立检查自己的维度
  ⚠️ 盲测模式: 不读实现代码，只看产品表现
     ↓
Simulator 团队 (并行):
  每个用户画像独立使用产品
  L1 可操控平台: 真实浏览器操作
  L2 不可操控平台: 截图/录屏分析
     ↓
反馈汇总 → 验收报告
```

**盲测的关键设计**:
- Guardian Agent 的 prompt 明确要求：只基于"用户能看到/感受到的"做判断
- 不读取源代码、不读取 API 文档、不读取数据库结构
- 如果某功能无法从用户视角验证（如后台算法），标记为"需人工确认"

**盲测隔离机制（核心）**:

orchestrator 在验收阶段必须以**独立子 Agent** 方式启动每个 Guardian 和 Simulator：

```
orchestrator (主会话，有完整项目上下文)
     ↓
  创建独立子 Agent（全新上下文，无项目记忆）
     ↓
  子 Agent 只接收:
    - 产品访问地址（URL/截图/录屏）
    - 验收标准清单（来自阶段一的输出）
    - 当前用户画像（Simulator 专用）
  ❌ 子 Agent 不接收:
    - 任何源代码
    - API 文档/数据库结构
    - 开发过程的讨论和决策
    - 已知的技术限制
```

**为什么必须用子 Agent**: 如果 orchestrator 自己验收，它带着"我是怎么做的"的记忆，天然会为自己的设计辩护。独立子 Agent 没有这些记忆——它看到的就是用户看到的，这才是真盲测。

### 5.4 阶段四：上线后 — 体验回溯

```
输入: 真实用户反馈 + 数据埋点
     ↓
feedback-compiler:
  1. 对照验收报告 → 哪些预估问题真的发生了？
  2. 对照模拟反馈 → 模拟用户的判断和真实用户一致吗？
  3. 校准 Agent → 提升后续项目的判断准确率
  4. 冲突裁决 → Guardian 和 Simulator 结论矛盾时的处理（见下方）
     ↓
输出:
  - 迭代建议清单（按优先级排序）
  - Agent 校准记录
  - 冲突裁决记录

### 5.4.1 冲突裁决规则

当 Guardian 和 Simulator 对同一功能给出矛盾结论时，按以下规则裁决：

```
规则 1: Simulator 发现的问题 Guardian 没发现
  → Simulator 胜。标记为"Guardian 漏检"，更新 Guardian 检查标准

规则 2: Guardian 说通过但 Simulator 用户实际卡住了
  → Simulator 胜。真实用户不会骗人。降低该维度 Guardian 可信度权重

规则 3: Guardian 说有问题但所有 Simulator 用户都没遇到
  → Guardian 胜。标记为"理论风险，实际未触发"，降低优先级

规则 4: 两个系统结论一致
  → 确认问题/确认通过。最高置信度
```

**核心原则**: **Simulator 是 Guardian 的验证者。Guardian 说没问题 ≠ 真的没问题；Simulator 说有问题 = 一定有问题。** 最终验收报告以 Simulator 的真实使用反馈为最高优先级。
```

## 6. 三级真实模拟体系（User Simulator）

### 6.1 L1: 直接操控（真实度 ⭐⭐⭐⭐⭐）

Agent 通过 Chrome MCP 工具直接操作浏览器：
- 打开网页/小程序 Web 版
- 真实点击、输入、滚动、等待
- 观察页面变化、加载状态、错误提示
- 记录每一步的"用户感受"

**适用**: Web 应用、PWA、H5 页面、小程序 Web 版

### 6.2 L2: 视觉分析（真实度 ⭐⭐⭐⭐）

产品方提供实际运行截图或录屏：
- Agent 逐帧分析界面和交互
- 识别视觉层次、间距、色彩、动效
- 判断信息架构和导航逻辑
- 标注体验问题位置

**适用**: 原生 App、微信小程序（真实环境）、游戏

### 6.3 L3: 测试剧本（真实度 ⭐⭐⭐）

Agent 生成详细的用户使用剧本：
- 场景描述 + 操作步骤 + 观察要点
- 真人按剧本使用产品后反馈
- Agent 分析反馈 → 提炼问题

**适用**: 所有平台的兜底方案、需要主观判断的复杂场景

### 6.4 自动分级决策

orchestrator 自动判断使用哪一级：

```
检测项目类型
  ├── 有 Web 版本? → L1 优先
  ├── 纯原生/小程序? → L2 优先
  └── L1/L2 都不可用? → L3 兜底
```

## 7. 用户画像系统

### 7.1 8 类基础画像模板

| 画像 | 特征 | 关注重点 |
|------|------|---------|
| **新手小白** | 第一次使用，技术不熟练 | 是否容易上手、引导是否清晰 |
| **效率专家** | 高频使用，追求效率 | 操作是否快捷、是否有批量能力 |
| **暴躁用户** | 耐心有限，容易放弃 | 响应速度、错误恢复、容错性 |
| **佛系用户** | 慢慢探索，非目标导向 | 发现的乐趣、内容深度 |
| **完美主义** | 关注细节，追求精致 | 设计还原度、动效品质、一致性 |
| **特殊需求** | 视力障碍/色盲/老年用户 | 可访问性、字体大小、对比度 |
| **恶劣环境** | 弱网/低端设备/小屏幕 | 降级体验、离线能力、适配 |
| **恶意用户** | 故意输入异常数据 | 输入校验、安全边界、防注入 |

### 7.2 动态画像生成

persona-generator 基于项目类型，从 8 类基础画像中动态组合出 3-5 个该产品的具体用户画像。

例如：电商小程序 → 「新手下单用户」「高频比价用户」「弱网场景用户」「老年大字体用户」

## 8. 输出物设计

### 8.1 验收报告

```markdown
# [项目名] 验收报告
生成时间: 2026-xx-xx | 阶段: 开发后盲测验收

## 总体评分: 82/100

## 各维度评分
| 维度 | 得分 | 状态 |
|------|------|------|
| 可达性 | 85 | ✅ 通过 |
| 可理解性 | 78 | ⚠️ 需改进 |
| 可靠性 | 90 | ✅ 通过 |
| 可响应性 | 72 | ⚠️ 需改进 |
| 愉悦性 | 88 | ✅ 通过 |
| 包容性 | 70 | ⚠️ 需改进 |

## 详细检查结果
[每个维度展开: 检查项 → 结果 → 证据/截图 → 建议]

## 用户模拟反馈汇总
[3-5 个画像的使用体验摘要 + 关键问题]

## 迭代建议（按优先级排序）
1. [P0] ...
2. [P1] ...
```

### 8.2 用户体验日志

每个模拟用户画像独立产出一份体验日志，记录使用过程中的：
- 目标 → 操作路径 → 遇到的困难 → 情绪变化 → 最终感受

### 8.3 迭代建议清单

按优先级排序的具体改进建议，每项包含：
- 问题描述
- 影响范围（哪些用户画像受影响）
- 严重程度
- 建议方案

### 8.4 报告存储规范

所有报告统一输出到项目根目录下的 `.quality-guardian/reports/`，按阶段和时间戳命名：

```
.quality-guardian/reports/
├── 2026-07-22-pre-acceptance-standard.md    ← 阶段一: 开发前验收标准
├── 2026-07-25-check-incremental.md          ← 阶段二: 开发中增量检查
├── 2026-07-30-acceptance-report.md          ← 阶段三: 开发后盲测验收
├── 2026-07-30-persona-新手小白-log.md        ← Simulator 各画像体验日志
├── 2026-07-30-persona-效率专家-log.md
├── 2026-08-05-review-retrospective.md       ← 阶段四: 上线后回溯
└── history.json                             ← 所有报告索引 + 分数变化趋势
```

**对比机制**:
- 阶段三的验收报告中自动包含「与开发前标准的对比」表格
- `history.json` 记录每次验收的各维度得分，支持趋势追踪
- 同一项目多次验收之间可定位：哪些问题修了、哪些新增了、哪些反复出现

## 9. 评分方法论

### 9.1 单检查项评分（4 级量表）

每个检查项采用统一的 4 级评分：

| 分数 | 等级 | 含义 | 示例 |
|------|------|------|------|
| **0** | 🔴 缺失/严重 | 用户无法完成核心任务 | 登录按钮点击无反应、支付流程中断 |
| **1** | 🟡 存在但有问题 | 用户能完成但费力或困惑 | 找到了按钮但位置不直观、错误提示是英文 |
| **2** | 🟢 合格 | 用户顺利完成，无阻碍 | 操作流程顺畅、信息清晰可理解 |
| **3** | 🟣 超出预期 | 用户感到惊喜或愉悦 | 智能预填、贴心的微交互、超出预期的体验细节 |

### 9.2 维度得分计算

```
维度得分 = (该维度所有检查项得分之和) / (检查项数量 × 3) × 100
```

示例：可达性维度有 10 个检查项，得分分别为 2,2,3,2,1,2,2,3,2,2
→ 维度得分 = (2+2+3+2+1+2+2+3+2+2) / (10×3) × 100 = 21/30 × 100 = **70**

### 9.3 总体得分计算

```
总体得分 = 6 维度得分加权平均

权重（可调整）:
  可达性:     15%
  可理解性:   20%  ← 对用户体验影响最大
  可靠性:     25%  ← 底线，权重最高
  响应性:     15%
  愉悦性:     15%
  包容性:     10%
```

### 9.4 评分门槛

| 分数 | 状态 | 含义 |
|------|------|------|
| ≥ 85 | ✅ 通过 | 该维度品质达标 |
| 70-84 | ⚠️ 需改进 | 存在问题，建议修复后再上线 |
| < 70 | ❌ 不通过 | 存在严重问题，必须修复 |

### 9.5 评分原则

- **证据驱动**: 每项评分必须附带截图/操作记录/用户反馈作为证据
- **用户视角**: 所有评分基于"用户实际体验"，不基于"技术上做得怎么样"
- **一致性**: 同一项目多次验收使用相同标准，确保可比性

## 10. 使用方式

### 10.1 安装（Claude Code）

```bash
# 克隆到项目的 .claude 目录
git clone https://github.com/quality-guardian/quality-guardian.git .claude/quality-guardian
```

### 10.2 使用

在 Claude Code 中直接调用：

```
/quality-guardian           # 自动检测项目，运行完整流程
/quality-guardian pre       # 仅开发前：生成验收标准
/quality-guardian check     # 仅开发中：增量检查
/quality-guardian accept    # 仅开发后：盲测验收
```
